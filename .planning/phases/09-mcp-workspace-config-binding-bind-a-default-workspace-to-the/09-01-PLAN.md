<objective>
Add the per-connection default-workspace mechanism: an HTTP middleware reads the `?workspace=` URL query param, injects it into the request `context.Context`, and `requireWorkspace`/`requireRegisteredWorkspace` use it as a fallback source when a tool call omits the `workspace` argument.

Purpose: This is the core mechanism the whole phase rests on (D-01, D-03, D-04, D-05, D-07). Without it, the schema changes in Plan 02 and the tests in Plan 03 have nothing to exercise. It reuses `storage.ResolveWorkspaceParam` unchanged — no parallel resolution path.
Output: The middleware, context key, and the two modified validation functions, plus unit tests proving the arg > context-default > error precedence.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| MCP client (`.mcp.json`) → HTTP server | The `?workspace=` value on the connection URL is attacker-controlled-if-the-config-file-is; `.mcp.json` is a locally-trusted file per this project's existing localhost/PNA threat model (RESEARCH.md Security Domain). |
| HTTP middleware → SDK tool dispatch | The injected context value crosses into the SDK's `req.Context()`, then into every tool handler's `ctx`. |
| Tool handler → storage resolution | `requireWorkspace`/`requireRegisteredWorkspace` pass the value (arg OR context default) into `storage.ResolveWorkspaceParam` and, for writes, the `GetWorkspaceByHash` registration check. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-09-01 | Elevation of Privilege | requireRegisteredWorkspace (write path) | high | mitigate | The connection-default value flows through the SAME `requireWorkspace` -> `ResolveWorkspaceParam` -> `GetWorkspaceByHash` registration path as an explicit arg (Task 2 restructure delegates fully to requireWorkspace; no laxer path is introduced). A query-param default can never grant write access beyond what the same string as an explicit arg would grant. Verified by Plan 03's write-path security test. |
| T-09-02 | Spoofing / wrong-workspace resolution | requireWorkspace fallback | high | mitigate | The `?workspace=` value is resolved by the identical `storage.ResolveWorkspaceParam` used for explicit args today — no second resolution path (RESEARCH.md V5). A malformed/malicious value fails cleanly with the existing "workspace not found"/prefix-ambiguity errors; it cannot silently resolve to a different valid workspace because resolution is deterministic (exact hash / unique prefix / exact name). |
| T-09-03 | Information Disclosure | requireWorkspace fallback (D-03 precedence) | high | mitigate | An explicit per-call `workspace` arg is read FIRST and always wins (Task 2: arg read before context lookup); a stale/compromised context default can never override an explicit arg, so a compromised connection default cannot redirect an explicitly-scoped call to the wrong workspace. |
| T-09-SC | Tampering | npm/pip/cargo installs | n/a | accept | No package installs in this phase (RESEARCH.md Package Legitimacy Audit: not applicable — zero new dependencies). No slopcheck checkpoint required. |
</threat_model>
