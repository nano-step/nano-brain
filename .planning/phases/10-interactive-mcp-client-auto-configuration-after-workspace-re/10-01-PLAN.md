<objective>
Add the workspace `name` field to the `POST /api/v1/init` response contract and decode it in the CLI, so downstream config-writers (Plan 02) can build a human-readable `?workspace=<name>` MCP binding instead of a raw hash.

Purpose: Per RESEARCH.md Pitfall 3, `name` (computed server-side as `filepath.Base(absPath)` at workspace.go:129, persisted via `UpsertWorkspace`) is currently NEVER returned to the CLI — `initResponse` only carries `WorkspaceHash`/`RootPath`/`AgentsSnippet`. Phase 9's `?workspace=` contract (D-03) wants the friendly name. This is the single most load-bearing prerequisite; it must land before any client-config writer can produce a name-bound URL. The anti-pattern of recomputing `filepath.Base(root)` client-side is explicitly forbidden (RESEARCH Anti-Patterns) — the server-side `name` is the canonical value and future server logic (sanitization, collision handling, renaming) could diverge from the CLI's guess.
Output: An extended response DTO (server) + matching decode struct (CLI), both covered by tests.
</objective>
