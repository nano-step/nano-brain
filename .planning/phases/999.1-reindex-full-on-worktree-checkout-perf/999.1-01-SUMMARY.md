# Phase 999.1 Plan 01: Reorder processFile content-hash dedup before edge extraction Summary

Pure statement-reorder in `processFile` (D-06a): `GetDocumentBySourcePath` hash-match check moved to run immediately after `contentHash` is computed and before the `w.graphRegistry != nil` edge-extraction block, eliminating redundant `extractAndUpsertEdges` calls for byte-identical files.

## What Was Built

### Task 1 + 2 (TDD RED→GREEN)

**`internal/watcher/watcher_test.go`** — added:
- `mockGraphQuerier` struct implementing `GraphQuerier` (`UpsertGraphEdge`, `DeleteGraphEdgesByFile`) with `atomic.Int64` counters.
- `TestProcessFile_ContentHashSkipsEdgeExtraction`: pre-seeds the mock querier with the file's content hash (simulating a file already indexed in a prior run), clears the mtime cache, then calls `processFile`. With `db==nil`, reaching `extractAndUpsertEdges` would panic at `w.db.BeginTx` — that panic is the RED signal. After the reorder, the hash-match early return fires first and the test passes cleanly.

**`internal/watcher/watcher.go`** — `processFile` reordered (lines ~749–762):

Before (incorrect order):
```
extractAndUpsertEdges(...)   ← ran first for every file
GetDocumentBySourcePath(...)  ← dedup checked second
if hash matches → return
```

After (correct order, D-06a):
```
GetDocumentBySourcePath(...)  ← dedup checked first
if hash matches → repopulate cache → return
extractAndUpsertEdges(...)   ← only reached on hash miss
```

No other statements, conditions, or behaviour changed.
