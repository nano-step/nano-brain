# Phase 999.1: Avoid full reindex on git worktree create / checkout - Context

**Gathered:** 2026-06-29
**Status:** Ready for planning (BACKLOG — not for immediate fix)

<domain>
## Phase Boundary

Reduce indexing lag on three triggers that share one root cause:
1. **`serve` startup** — machine lags for a while right after starting nano-brain (user-reported: "thấy index hoặc chunks rất nhiều").
2. **New git worktree** — creating a worktree triggers the same lag.
3. **Git checkout / branch switch** — rewrites file mtimes, defeating the cheap skip.

This phase clarifies WHAT to fix and HOW we'd verify it. It does NOT implement the fix yet.

**Out of scope:** changing the embedding pipeline, search/ranking, or chunking logic. This is strictly about *deciding what to re-examine* on these triggers.
</domain>

<decisions>
## Implementation Decisions

### Reframing — the original assumption was partly wrong (D-01)
- **D-01:** Content-hash dedup ALREADY EXISTS at two levels — document (`internal/watcher/watcher.go:757`) and chunk (`internal/db/chunks.sql.go:174-177`). A checkout/worktree with identical content does **NOT** re-embed. So the lag is **not** redundant embedding — it is the full filesystem scan + per-file SHA256 + per-file DB lookup that runs because the cheap skip is defeated.

### Root cause (D-02)
- **D-02:** The cheap skip — mtime+size cache (`fileState`, `watcher.go:62-66`, checked at `watcher.go:722`) — is **in-memory only**. It is empty on every process start and for every new worktree, so every file falls through to Stage-2 hashing. Git checkout additionally rewrites mtimes, defeating the cache even within a long-running process. This one fact explains all three symptoms.

### Scope (D-03)
- **D-03:** Address all three triggers together — they share the root cause (non-persistent cheap-skip fingerprint + zero git-awareness in the watcher). Startup-serve lag is the most frequent and was the user's strongest signal.

### Verify-first gate (D-05) — SATISFIED 2026-06-30
- **D-05:** Gate met by direct evidence, not a separate benchmark: live daemon logs show a `graph edges extracted component=watcher edges=N file=...` storm on every `serve` startup, and a code trace pinned the exact mechanism (below). Embedding is NOT re-triggered (chunk dedup holds); the lag is **graph/symbol re-extraction (tree-sitter) on every unchanged file**. No further measurement needed before fixing.

### Pinpointed root cause (confirmed)
In `internal/watcher/watcher.go` `processFile`:
1. Fast-path skip (`:717-724`) reads `w.fileCache` — an **in-memory** `map[string]fileState` (`:103,:137`) — so it is empty on every startup → no file is skipped.
2. **Ordering bug:** `extractAndUpsertEdges` (`:749-751`) runs BEFORE the content-hash dedup check (`:753-757`). So even byte-identical files get tree-sitter edges re-extracted on every startup. This is the observed log storm.

Prior art: issue #375 / PR #388 added the in-memory mtime+size+hash cache (the `:717` fast-path) deliberately as in-memory — fixes intra-process rescans but NOT restarts. That is exactly the gap.

### Locked solution (D-06) — implement both, via GSD
- **D-06a (fix A — reorder, cheap):** Move the content-hash dedup check (`:753-757`) to run BEFORE `extractAndUpsertEdges` (`:749`). Byte-identical content ⇒ identical edges ⇒ skip extraction. Also helps the git-checkout case (mtime differs → falls through → hash matches → skip edges).
- **D-06b (fix B — persist fast-path):** Make the `:717` mtime+size fast-path survive restart by **warming `w.fileCache` from the DB at watcher startup** (minimal — reuse existing skip logic, no rewrite):
  - Migration: add `mod_time TIMESTAMPTZ` + `file_size BIGINT` to `documents`.
  - Populate both in `UpsertDocumentBySourcePath` (from `os.Stat` at index time).
  - On watcher start: one bulk query `(source_path → mod_time, file_size, content_hash)` for the workspace → preload into `w.fileCache`.
- **Result:** unchanged file on a plain restart → skipped with no read/hash/extraction. Fixes the serve-startup symptom (the primary complaint).

### Known caveat (locked, documented)
- Git **checkout rewrites mtime** → B's fast-path won't match → file falls through to read+hash; fix A then skips edges via the hash match. So B fully fixes restart; A covers checkout. Full git-diff awareness (old D-04c) is explicitly **deferred** — not needed once A+B land.

### Claude's Discretion
- Exact migration number, column nullability/backfill strategy, and the preload query shape.
- Whether the preload is eager at startup or lazy per-collection on first scan.

<code_context>
## Existing Code Insights

### Reusable Assets
- Content hashing + `GetDocumentBySourcePath` already exist — a persisted fingerprint can reuse the same hash, just compared earlier/cheaper.
- The two-stage skip architecture is already in place; the fix is making Stage-1 durable, not adding a new stage.

### Established Patterns
- Stage-1 (cheap mtime+size) → Stage-2 (content hash) → embed. Preserve this ordering; the change is persisting Stage-1 state.
- Incremental reindex handler already walks disk and compares `content_hash` — a model for the "examine only what changed" logic.

### Integration Points
- DB schema (sqlc — `sqlc generate` after SQL changes) for the persisted fingerprint column.
- Watcher startup (`processAll`) and worktree/workspace resolution are where a persisted fingerprint would be loaded.
</code_context>
