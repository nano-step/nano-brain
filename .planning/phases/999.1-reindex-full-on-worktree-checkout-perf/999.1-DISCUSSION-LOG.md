## Scope

| Option | Description | Selected |
|--------|-------------|----------|
| Git checkout (existing workspace) | mtime rewrite defeats cheap skip → full hash scan | ✓ |
| New worktree | fresh workspace, no cache/DB → hash (maybe embed) from scratch | ✓ |
| Both | shared root cause, different reuse strategy | ✓ |

**User's choice:** Both — and revealed a third trigger: **`serve` startup** also lags ("khi start nano-brain serve lên là thấy index hoặc chunks rất nhiều ... máy lag một thời gian"). Asked Claude to debug the real trigger.
**Notes:** Scout + this signal converge on one root cause: the cheap mtime+size skip cache is in-memory only, so it is empty on every startup and every new worktree, and is defeated by git's mtime rewrite on checkout.

---

## Solution direction

| Option | Description | Selected |
|--------|-------------|----------|
| Git-aware diff | detect HEAD change, `git diff --name-only` only changed files | (kept as researched option) |
| Persist mtime/size to DB | durable cheap fingerprint surviving restart | ✓ |
| Reuse sibling worktree index | copy/share content_hash for same commit | (folded into research) |
| Speed up current scan | parallelize hashing + batch DB lookups | (fallback only) |

**User's choice:** Persist fingerprint to DB — with explicit instruction to "research các tool khác đang làm thế nào và làm theo" (study how mature tools do it and follow the proven pattern).
**Notes:** User is unsure of the exact fingerprint scheme; delegated to research. Watchman, ctags/gtags, Sourcegraph/zoekt, ripgrep, LSP/tree-sitter indexers, git's stat cache identified as study targets.
