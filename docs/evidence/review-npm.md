## Findings — resolved

- **BLOCKER (resolved)** — `npm/postinstall.js:24-31` (`getPlatformKey`). Now `throw new Error("Unsupported platform: ...")` instead of `process.exit(0)`, so `ensureBinary()`'s "throws, never exits" contract actually holds. `main()` (postinstall.js:341-357) catches the non-`SECURITY:` error → `process.exit(0)` (npm install never fails), and `run.js`'s `.catch` (run.js:48-53) reports it and exits 1 — restoring the pre-PR exit-1-on-missing-binary behavior. Repro (os.platform stubbed to freebsd, no network): the line *after* the `ensureBinary()` call — `reachable-after-call` — now prints, then `REJECTED: Unsupported platform: freebsd-arm64` fires via the promise chain. Under the old code that line was unreachable because `process.exit()` fired synchronously. New test `ensureBinary: rejects (does not exit) on unsupported platform` (postinstall.test.js:32-41) locks this in with `assert.rejects(..., /Unsupported platform/)`.

- **WARNING (resolved, stdout)** — `tryAutoLink` is removed from `attempt()`/`ensureBinary()` and now called ONLY from `main()` (postinstall.js:348), i.e. install time. Grep confirms exactly one call site outside the definition (line 348). `run.js`'s lazy-download path invokes `ensureBinary()`, which no longer emits any auto-link `console.log` chatter, so a lazy `nano-brain version --json` keeps stdout clean. Verified end-to-end: a `NANO_BRAIN_BIN`-stub `run.js` invocation produced stdout = `NB_STUB_OK` only, stderr empty. Progress/warn lines (`Downloading…`, `⚠ NANO_BRAIN_SKIP_SHA_VERIFY`, `installed successfully`) all correctly go to `console.error` (stderr).

- **WARNING (resolved, run.js structure)** — `npm/run.js:33-38` now has an explicit `return` after each terminal branch (`NANO_BRAIN_BIN` override at :33-34, existing-binary fast path at :37). Mutual exclusivity is now structural rather than relying on `runBinary()` always exiting. Top-level `return` is valid here (CommonJS module wrapper) — confirmed by executing the file; the three branches (override → existing → lazy-download) are correctly ordered, each terminal, no fall-through.

- **WARNING (resolved, tests)** — the fast-path test (postinstall.test.js:15-30) no longer silently no-ops: it renames any real binary aside to `.itest.bak`, writes the stub, asserts, then restores in `finally`, so the assertion always runs. Added: the unsupported-platform reject test, and a real `run.js` subprocess test (postinstall.test.js:46-60) that spawns `run.js` with `NANO_BRAIN_BIN` pointing at a stub and asserts `status === 0` + stdout marker — the first coverage of run.js's exec path. ## Residual (non-blocking, accept)

- **Behavior nuance:** the old `main()` only called `tryAutoLink` immediately after a *fresh download*; the new `main()` calls it on every successful `ensureBinary()` return, including the "binary already present and valid" fast path. For opt-in users (`NANO_BRAIN_AUTO_LINK`/marker) this simply means auto-link is re-attempted on re-install; `tryAutoLink` already no-ops when the target exists (postinstall.js:255-258). Install-time only (no stdout concern) and arguably more correct. Not a regression.
- **Concurrency (acknowledged, pre-existing):** `download`/`downloadWithHash` write directly to `binaryPath()` with no atomic temp+rename; two concurrent lazy `run.js` invocations sharing a `node_modules` could interleave. Predates this PR; the lazy path makes it marginally more reachable. Correctly deferred as a follow-up per prior review.
- **Lazy-download success path** could not be exercised locally (package version is `0.0.0-dev`; no matching release asset → HTTP 404), but the coordinator's e2e (real install, binary absent → `run.js version` → `nano-brain v2026.7.1103` on stdout, progress on stderr, exit 0; and a network-fail run → clean stderr + exit 1, no crash) plus the structural tests cover the contract.
- **CI note (unchanged):** these npm tests are still not wired into `.github/workflows/ci.yml` (Go-only pipeline), so the suite runs manually only. Non-blocking, but wiring `node --test npm/` into CI would be a sensible follow-up.
