Feature deep-dive

Under the hood

Six subsystems. All running in the background. All adapting over time.

01

Search

Hybrid search pipeline

Every query passes through up to 11 ranked stages before returning results. The blend weights are not hardcoded — they are tuned continuously by Thompson Sampling bandits using retrieval telemetry.

BM25 handles exact-match and keyword recall via SQLite FTS5. Vector search handles semantic similarity via 768-dim embeddings in Qdrant. RRF fusion (k=60) blends both scores. PageRank adds a centrality boost from the file dependency graph. Supersede demotion deprioritizes stale documents. VoyageAI neural reranking (rerank-2.5-lite) makes the final pass.

1 BM25 / FTS5 SQLite · exact match
2 Vector similarity Qdrant · cosine
3 RRF fusion k=60 · blends 1+2
4 Top rank bonus FTS top-3 boost
5 Centrality boost PageRank from dep graph
6 Usage boost Retrieval frequency
7 Category weight Preference learning
8 Supersede demotion Stale doc filter
9 Length normalization Penalize short chunks
10 Recency boost Favor recent sessions
11 Neural reranking Cohere / VoyageAI (optional)

Weights tuned by Thompson Sampling from search telemetry

02

Code Intelligence

AST-level code understanding

Tree-sitter parses TypeScript, JavaScript, and Python source files into ASTs. nano-brain extracts symbol definitions, builds a call graph (caller→callee edges), and maps the file dependency graph (imports/exports).

These graphs power three MCP tools: code_context gives a 360° view of any symbol; code_impact does BFS traversal to find everything affected by a change; code_detect_changes maps a git diff to affected symbols in seconds.

04

Storage

SQLite + Qdrant, zero config

All persistence is SQLite (better-sqlite3, WAL mode). Each workspace gets its own database file under ~/.nano-brain/data/, keyed by a 12-char SHA-256 of the workspace path.

Vectors are stored in Qdrant (768-dim collections). Content is content-addressed — documents are deduplicated by SHA-256, so re-indexing is always safe and idempotent.

The 18-table schema covers documents, chunks, FTS5 index, symbol graph, entity graph, telemetry, and bandit variants.

Table Purpose
documentsContent-addressed document metadata
chunksHeading-aware text chunks (900 tok)
contentRaw content (SHA-256 dedup)

bandit_variantsThompson Sampling state
05

Memory Intelligence v2

Memory that improves itself

Three adaptive layers make nano-brain smarter the more you use it.

Entity Pruning

Automatically soft-deletes contradicted entities (30 days TTL) and orphan entities (90 days TTL). Hard-deletes after 30-day retention. Prevents the knowledge graph from accumulating stale information.

6h interval batch 100

LLM Categorization

After every memory_write, a background LLM call assigns llm: category tags. Fire-and-forget, non-blocking. Makes memories discoverable by topic without manual tagging.

async llm: tags

Preference Learning

Thompson Sampling bandits track which content types you retrieve most. RRF blend weights and reranking thresholds adapt over time. Every search trains the model; every session makes the next one better.

Thompson Sampling 10m cycle
06

Workspace isolation

One DB per project, enforced

Every workspace path gets its own SQLite database named {dir}-{hash12}.sqlite. Since v2026.8.19, the server validates --root against config.workspaces — unconfigured paths fall back to the closest configured workspace instead of creating a new orphaned database. 

# ~/.nano-brain/config.yml
workspaces:
  /Users/you/projects/my-app:
    codebase:
      enabled: true
      extensions: [.ts, .tsx, .py]
  /Users/you/projects/api:
    codebase:
      enabled: true
      maxFileSize: 200KB

# Resulting databases in ~/.nano-brain/data/
