# Epic 4: Hybrid Search — User Stories

**Epic summary:** Full hybrid search pipeline: BM25 + vector search in parallel, RRF fusion,
recency boost. Three search endpoints with consistent schema. Wake-up briefing. Search completes
in under 3 seconds at scale.

**FRs covered:** FR-7, FR-8, FR-10, FR-11, FR-12, FR-13, FR-14, FR-15, FR-56, FR-57, FR-60

**ARs applied:** AR-8 (sqlc for all BM25 queries)

**NFRs enforced:**
- NFR-2: Workspace isolation in every query — WHERE clause, not post-filter
- NFR-3: Search quality >= v1 (P@5>=0.835, R@10>=0.970, MRR>=1.000)

**Dependency note:** These stories assume Epic 3 is complete (embeddings exist in the `embeddings`
table) and Epic 2 is complete (documents and chunks are in the database with BM25 tsvectors indexed).

## Stories

### Story 4.1: BM25 Search Endpoint

**Description:** Implement `POST /api/search` — the BM25-only search path. This is the foundation
that hybrid search builds on top of. It uses PostgreSQL `tsvector`/`tsquery` to match keyword
queries against indexed chunks, returns results in the canonical schema, and enforces workspace
isolation at the query layer via sqlc-generated code.

**Covers:** FR-8, FR-10, FR-11, FR-12, FR-57

**Applies:** AR-8

**Complexity:** M

**Acceptance Criteria:**

- Given a workspace with indexed documents, when `POST /api/search` is called with `{"query": "database migration", "workspace": "<hash>"}`, then it returns `{"results": [...], "total": N, "query_ms": N}` with HTTP 200.
- Given results are returned, when each result is inspected, then it contains all required fields: `id`, `title`, `snippet`, `score`, `tags`, `collection`, `workspace_hash`, `created_at`, `updated_at`.
- Given a query with `workspace` omitted, when `POST /api/search` is called, then it returns HTTP 400 with `{"error": "workspace_required", "message": "..."}`.
- Given workspace A has document D and workspace B does not, when `POST /api/search` is called scoped to workspace B, then document D is NOT returned. (Workspace isolation negative test — isolation enforced via WHERE clause, not post-filter.)
- Given the BM25 query SQL is generated by sqlc, when the generated query is reviewed, then it includes `WHERE workspace_hash = $1` as a required parameter, not an optional filter.
- Given `limit` is provided in the request, when results are returned, then the count does not exceed `limit`. Default limit is 10 when omitted.
- Given `tags` are provided in the request, when results are returned, then all results have at least one matching tag.
- Given a non-JSON body, when `POST /api/search` is called, then it returns HTTP 415.
- Given a query, when the response is returned, then the `X-Nano-Brain-Version` header is present.

**Test expectations:**
- Unit: sqlc query compiles and includes workspace_hash param; RRF/scoring logic tested in isolation.
- Integration (real PG, `//go:build integration`): insert docs into workspace A and workspace B, query workspace A, assert workspace B documents never appear. Run 100 queries with mixed workspaces and assert zero cross-workspace leaks.
- Performance: BM25 search on 100K documents completes in under 200ms p50, under 500ms p95.

### Story 4.2: Consistent Result Schema Across All Search Endpoints

**Description:** All three search endpoints (`/api/query`, `/api/search`, `/api/vsearch`) must
return the same JSON schema. This story defines and validates the canonical schema structure,
ensuring downstream consumers (MCP tools, CLI) can treat responses uniformly without endpoint-specific
parsing. It covers the shared response type, snippet generation (max 700 chars), and score
normalization contract.

**Covers:** FR-10

**Applies:** AR-8

**Complexity:** S

**Acceptance Criteria:**

- Given any of the three search endpoints returns a response, when the schema is validated, then it matches exactly: `{results: [{id, title, snippet, score, tags, collection, workspace_hash, created_at, updated_at}], total, query_ms}`.
- Given a result is returned, when `snippet` is inspected, then it is at most 700 characters.
- Given a result is returned, when `score` is inspected, then it is a non-negative float (scores are not normalized to 0-1 in MVP, but are stable across identical queries).
- Given `tags` is null or absent for a document, when the result is serialized, then `tags` is omitted from the JSON (not sent as `null`), per `json:",omitempty"` convention.
- Given `total` in the response, when compared to `len(results)`, then `total` reflects the total number of matching documents found before `limit` was applied, not just the returned page size.
- Given the same query is run twice on an unchanged index, when results are compared, then ranking order is identical (deterministic scoring).

**Test expectations:**
- Unit: shared result type serializes correctly; `omitempty` on nil tags; snippet truncation at 700 chars.
- Integration: call all three endpoints with the same query, deserialize all three responses, assert identical schema shape.

### Story 4.3: 7-Stage Hybrid Search Pipeline

**Description:** Implement the full hybrid pipeline behind `POST /api/query`. The pipeline runs
BM25 search and vector search in parallel (via goroutines), fuses the ranked lists with RRF
(k=60, configurable), applies recency boost (weight=0.3, half-life=180 days, both configurable),
selects the top-K results, and formats output. This is the primary search path agents use.

**Covers:** FR-7, FR-11, FR-12, FR-13, FR-14, FR-15, FR-56

**Applies:** AR-8

**Complexity:** L

**Acceptance Criteria:**

- Given a workspace with documents that have embeddings, when `POST /api/query` is called with `{"query": "session context", "workspace": "<hash>"}`, then it returns results from both BM25 and vector search paths, merged and ranked.
- Given BM25 finds documents D1, D2 and vector search finds D2, D3, when RRF fusion runs, then D2 ranks higher than it would from either signal alone (fusion reward).
- Given `search.rrf_k` is set to 30 in config, when RRF runs, then k=30 is used in the score formula `score = Σ weight / (k + rank + 1)`, not the default 60.
- Given a document was written 10 days ago and another 200 days ago, when recency boost is applied, then the newer document receives a higher multiplier, proportional to `exp(-ln(2) * age_days / half_life_days)`.
- Given `search.recency_weight = 0.3` (default), when the final score is computed, then it is `(1 - recency_weight) * rrf_score + recency_weight * recency_score`.
- Given `search.recency_weight` and `search.recency_half_life_days` are changed in config, when a search runs, then the new values take effect (reloadable via `POST /api/reload-config`).
- Given BM25 and vector stages run in parallel, when either stage fails (e.g., no embeddings), then the other stage's results are still returned (graceful degradation).
- Given `scope=all` in the request, when the query runs, then results from all registered workspaces are merged and returned. The query uses explicit workspace hash list, not `WHERE 1=1`.
- Given a workspace with 100,000 documents, when `POST /api/query` runs, then the response arrives within 3 seconds (FR-13).
- Given workspace A has document D and workspace B does not, when `POST /api/query` scoped to workspace A runs, then document D is returned. When scoped to workspace B, document D is NOT returned. (Isolation negative test.)

**Test expectations:**
- Unit: RRF math verified with known inputs (e.g., rank 1 and rank 2 lists merge to expected scores with k=60); recency multiplier formula verified with edge cases (very old docs, docs from today).
- Integration (real PG): populate two workspaces, run hybrid query on each, assert zero cross-workspace results across 500 permutations.
- Performance: 100K documents, single concurrent query, p50 < 200ms, p95 < 500ms, absolute max < 3000ms.

### Story 4.4: Workspace Isolation Invariant in Search

**Description:** The workspace isolation invariant is a binary release gate (G2): no query scoped
to workspace A can return a document from workspace B. This story is specifically about proving the
invariant at the search layer with automated tests that run against real PostgreSQL. It covers all
three endpoints, the `scope=all` multi-workspace path, and the HTTP 400 guard when workspace is
missing. The invariant must be enforced at the SQL query layer (WHERE clause), not by post-filtering
results in Go.

**Covers:** FR-11, FR-12

**Applies:** AR-8

**Complexity:** M

**Acceptance Criteria:**

- Given two workspaces W1 and W2 with non-overlapping document sets, when 1,000 hybrid queries are run alternating between workspaces, then zero documents from W1 appear in W2 results, and zero from W2 appear in W1 results.
- Given the sqlc query for hybrid search is reviewed, when the SQL is inspected, then `WHERE workspace_hash = $1` appears as a required positional parameter, not a conditional.
- Given `scope=all` is passed, when the query runs, then the implementation uses `WHERE workspace_hash = ANY($1::text[])` with an explicit list of hashes, not an unconstrained `WHERE 1=1`.
- Given `workspace` is missing from a search request body, when the handler processes it, then it returns HTTP 400 with `{"error": "workspace_required"}` before any database query executes.
- Given a write to workspace A is performed via `POST /api/write`, when `POST /api/query` immediately runs scoped to workspace B (with real PostgreSQL MVCC), then the new document is not returned.

**Test expectations:**
- Integration (`//go:build integration`, real PG): automated cross-workspace isolation suite. Create N workspaces, insert distinct documents, run all M*N workspace-pair query combinations, assert zero leaks. This is the G2 release gate test.
- This test MUST be in `internal/search/isolation_test.go` and tagged `//go:build integration`.

### Story 4.5: Configurable RRF and Recency Parameters

**Description:** RRF k and recency boost weight/half-life are configurable via `search.*` config
keys. They have sensible defaults but must be hot-reloadable. This story covers the config struct,
loader integration, default values, and validation. It also ensures these parameters flow correctly
into the search service constructor via dependency injection.

**Covers:** FR-14, FR-15

**Applies:** AR-8

**Complexity:** S

**Acceptance Criteria:**

- Given `search.rrf_k` is not set in config, when the server starts, then `rrf_k` defaults to 60.
- Given `search.recency_weight` is not set, when the server starts, then it defaults to 0.3.
- Given `search.recency_half_life_days` is not set, when the server starts, then it defaults to 180.
- Given `search.rrf_k` is set to 0 or a negative value in config, when the server starts, then it exits with a descriptive config validation error (not a panic at search time).
- Given `search.recency_weight` is set to a value outside [0, 1], when the server starts, then it exits with a config validation error.
- Given search parameters are changed in the YAML config file, when `POST /api/reload-config` is called, then subsequent searches use the new values without restart.
- Given a `SearchConfig` struct is defined in `internal/config/`, when `NewSearchService` is called, then it takes `SearchConfig` as a parameter (constructor injection, not global config access).

**Test expectations:**
- Unit: config defaults applied when keys absent; validation rejects out-of-range values; constructor receives correct values.
- Integration: change rrf_k via reload, run a search, assert the changed k is used in the RRF calculation (observable via score values or debug logging).

### Story 4.6: Wake-Up Briefing Endpoint

**Description:** `GET /api/wake-up` and `POST /api/wake-up` return a compact session briefing
for agents at session start. The briefing includes a summary, recent memories (last N documents
written), and active collections. This is what `memory_wake_up` wraps. It should return within
2 seconds and deliver enough context for an agent to resume work without re-reading raw logs.

**Covers:** FR-60

**Applies:** AR-8

**Complexity:** M

**Acceptance Criteria:**

- Given a workspace with recent documents, when `GET /api/wake-up?workspace=<hash>` is called, then it returns HTTP 200 with `{summary, recent_memories: [{id, snippet, tags, date}], active_collections: [...]}`.
- Given `POST /api/wake-up` is called with `{"workspace": "<hash>"}` in the body, then it returns the same schema as the GET variant.
- Given a workspace with 50 recent documents, when wake-up runs, then `recent_memories` contains the most recently updated documents (ordered by `updated_at DESC`), capped at a configurable limit (default 10).
- Given `workspace` is missing from both the query string (GET) and body (POST), when the endpoint is called, then it returns HTTP 400.
- Given a workspace with no documents, when wake-up runs, then `recent_memories` is an empty array and `active_collections` lists the collections (even if empty).
- Given a wake-up response on a workspace with 100 recent documents, when the response time is measured, then it completes in under 2 seconds (PRD §10 budget).
- Given `active_collections` in the response, when the list is compared to the workspace's registered collections, then it matches exactly (name, document count, last updated).
- Given the `summary` field, when it is returned, then it is a non-empty string that describes the workspace state (it may be generated from a template, not LLM; for MVP it is a structured summary of stats, not prose).

**Test expectations:**
- Unit: summary template generation; recent_memories query order and limit; empty workspace returns valid schema.
- Integration (real PG): insert 20 documents with known timestamps, call wake-up, assert most-recent 10 appear in `updated_at DESC` order.
- Performance: wake-up on workspace with 100 recent documents completes in under 2000ms p95.

## Implementation Notes

**Package:** `internal/search/` is the primary home for all stories. Key files:
- `service.go` — `SearchService` orchestrating the hybrid pipeline (Story 4.3)
- `bm25.go` — BM25 search via PG tsvector (Stories 4.1, 4.2)
- `rrf.go` — RRF merger, recency boost (Stories 4.3, 4.5)
- `interfaces.go` — `EmbeddingProvider` consumer-side interface (Story 4.3)

**HTTP handlers** live in `internal/server/handler/search.go` (Stories 4.1, 4.3, 4.6).

**sqlc queries** live in `internal/storage/queries/search.sql`. All queries MUST include
`WHERE workspace_hash = $1` — the AR-8 and NFR-2 gate.

**Parallelism pattern** for Story 4.3 (BM25 + vector in parallel):

```go
var bm25Results, vectorResults []SearchResult
g, ctx := errgroup.WithContext(ctx)
g.Go(func() error { bm25Results, err = s.bm25Search(ctx, req); return err })
g.Go(func() error { vectorResults, err = s.vectorSearch(ctx, req); return err })
if err := g.Wait(); err != nil { /* handle */ }
merged := s.rrf.Merge(bm25Results, vectorResults, cfg.RRFK)
boosted := s.applyRecencyBoost(merged, cfg.RecencyWeight, cfg.RecencyHalfLifeDays)
```

**Workspace isolation enforcement** is a hard rule across all stories. Any SQL in this epic that
touches `documents`, `chunks`, or `embeddings` tables MUST have `workspace_hash` as a required
(non-nullable) parameter, generated by sqlc, not added as an optional application-layer filter.
