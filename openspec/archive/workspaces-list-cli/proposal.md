# Workspaces List CLI + Status Count Fix

## Problem

Two related workspace-discoverability gaps:

1. **No CLI surface for listing workspaces.** After `nano-brain init --root <path>` succeeds, the only way to see registered workspaces is `curl http://localhost:3100/api/v1/workspaces`. The user must know the endpoint exists, the response shape, and have a JSON-friendly tool. `nano-brain config show` intentionally hides workspaces (they live in DB, not config — see existing design). `nano-brain doctor` doesn't list them either.

2. **`/api/v1/status` returns `workspace_count: 0` hardcoded.** Both `Health.Health()` (line 85) and `Health.Status()` (line 110) in `internal/server/handlers/health.go` set the count to a literal `0` instead of querying the DB. So `nano-brain status` always reports 0 workspaces even when many are registered.

Together these create the broken UX a user just hit: registered a workspace successfully, then had no idea where it went.

## Solution

### A. New CLI command `nano-brain workspaces`

Add a top-level subcommand mirroring the existing `nano-brain collection` pattern:

```
nano-brain workspaces list   # alias: ls
nano-brain workspaces        # short for `workspaces list`
```

- Calls `GET /api/v1/workspaces` (existing endpoint, no backend changes).
- Default output: human-readable table to stdout.
- `--json` flag: passthrough JSON exactly as the API returns.
- Empty result: prints `No workspaces registered.` to stderr + exit 0.

### B. Fix `/api/v1/status` and `/health` workspace count

Wire the existing `WorkspaceQuerier` (already has `ListWorkspacesWithStats`) into the `Health` handler. Use a lightweight count — a new sqlc query `CountWorkspaces` is preferred over re-using `ListWorkspacesWithStats` to avoid loading rows just to count.

## Scope

- **In scope**:
  - `cmd/nano-brain/workspaces.go` (NEW file, ~120 lines) — subcommand + table render + JSON passthrough
  - `cmd/nano-brain/main.go` — dispatch `case "workspaces"`
  - `internal/server/handlers/health.go` — replace hardcoded `0` with real query
  - `internal/storage/sqlc/queries/workspaces.sql` — add `CountWorkspaces` query
  - `internal/storage/sqlc/workspaces.sql.go` — regenerated by sqlc
  - `internal/server/handlers/health_test.go` — update mocks
  - `internal/server/routes.go` — pass querier to `NewHealth`
  - New tests: `cmd/nano-brain/workspaces_test.go`
- **Out of scope**:
  - Workspace deletion CLI (no backend `DELETE /api/v1/workspaces/:hash` exists yet — separate change)
  - Workspace rename (no backend support)
  - Per-workspace detail view (`workspaces show <hash>`) — defer
  - Sorting / filtering flags — defer (no user need yet)
  - Logging additions — owned by proposal #4 (issue #144)

- Multi-file change → 1 flag

- HTTP handler signature change (`NewHealth` gains a querier param) → 1 flag (internal API only, no public REST contract change)

**Total: 4 flags → upper edge of lane:normal.** No DB schema change, no public API contract break, no security/auth concern. Treating as `lane:normal` with extra Momus scrutiny on the handler-signature change.
